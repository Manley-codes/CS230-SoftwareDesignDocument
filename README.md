# CS-230 Software Design Document

[View the completed software design document](Final%20CS%20230%20Project%20Software%20Design.pdf)

The client for this project was The Gaming Room. They wanted help designing a web-based version of their game, Draw It or Lose It, so it could move beyond its original platform and support multiple operating platforms. The software needed to support multiple teams and players, keep game and team names unique, and use a structure where the main game logic could be managed from the server side. One thing I think I did well in developing the documentation was connecting the technical recommendations back to the client’s actual needs. Instead of only comparing Windows, macOS, Linux, and mobile platforms separately, I tried to think about which platform choice would make the most sense for the type of application The Gaming Room wanted to build. I also think I did well explaining why a client-server structure was a good fit, because the game needed to support multiple users and platforms while keeping the main logic and data organized.

Working through the design document was helpful because it made me slow down and think about the structure before jumping straight into the code. It helped me understand the purpose behind certain design choices, such as using the singleton pattern to make sure only one instance of the game service was created. It also helped me think through how the classes should relate to each other, how data should be managed, and how the system could support multiple players and teams. If I could revise one part of the document, I would improve the platform comparison and recommendation section by connecting each platform more directly to the client’s goals, especially around security, scalability, and long-term maintenance. I interpreted the user’s needs by focusing on what The Gaming Room was really trying to accomplish, not just what code needed to run. They needed a design that could support a real game environment with multiple teams, players, and unique names, so those needs guided the class structure and logic to prevent duplicates. My overall approach was to break the problem into smaller parts first, including the client’s requirements, users, platform needs, architecture, and server-side logic. In the future, I would use a similar process, but I would spend more time creating early diagrams, writing out user needs, and comparing design options before coding.
